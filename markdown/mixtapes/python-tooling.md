# Tooling Mix: the Python stack behind the demos

Every chapter in this library leans on the same small set of Python tools. They run the inline demos, check the proofs, and translate notation into something executable. This mix threads the best readings about them into one path — not a catalogue of links sorted by tool, but a sequence in which each reading prepares the ground for the next, because the tools themselves form a sequence: a way of running code leads to a way of writing mathematics as code, which leads to a way of asking a solver whether the mathematics is true.

## Side A — notation becomes computation

The path begins with the substrate. Everything in this repo runs through uv, and the page that explains the whole arrangement is [Running scripts](https://docs.astral.sh/uv/guides/scripts/): `uv run`, the inline `# /// script` dependency block, and `--with` for one-off imports. The commands scattered through these chapters stop looking magical after a single read. But the document that explains *why* the repo works this way is Simon Willison's [Building Python tools with a one-shot prompt](https://simonwillison.net/2024/Dec/19/one-shot-python-tools/) — his observation is that a script which declares its own dependencies is a different kind of artifact from a project, disposable yet reproducible, and that this changes what you reach for when a stray idea needs testing. Every demo in this library is that kind of artifact. If you want a map of uv's wider command surface before moving on, [First steps](https://docs.astral.sh/uv/getting-started/first-steps/) provides it in ten minutes.

With the substrate in place, the first real instrument is SymPy, because it makes the most direct version of this repo's claim: that notation can be executed. The [introductory tutorial](https://docs.sympy.org/latest/tutorials/intro-tutorial/intro.html) opens with exact computation — $\sqrt{8}$ becoming $2\sqrt{2}$ rather than $2.828$ — and that distinction between symbol and approximation is the whole idea of the library in miniature. Follow it immediately with [Gotchas and Pitfalls](https://docs.sympy.org/latest/tutorials/intro-tutorial/gotchas.html), which is really a lesson in representation: a `Symbol` is not a Python variable, `==` is not equality of expressions, and `1/2` is consumed by Python before SymPy ever sees it. Internalizing why those trip people up teaches you more about symbolic computation than the happy-path tutorial does.

SymPy executes the calculus notation; the summation notation belongs to einsum. The bridge is the same in both cases — a compact written form expands into loops — and reading einsum's subscript strings is the closest a programmer comes to reading Einstein's index notation natively. Start with Alex Riley's [basic introduction to einsum](https://ajcr.net/Basic-guide-to-einsum/), which builds the reading rules one operation at a time, then let Tim Rocktäschel's [Einsum is All You Need](https://rockt.ai/2018/04/30/einsum) show the payoff: dot product, outer product, matrix multiplication, and transpose all collapse into a single uniform notation, which is exactly the fluency that makes $\Sigma_i a_i b_i$ readable at a glance. When something doesn't behave, Eli Bendersky's [Understanding NumPy's einsum](https://eli.thegreenplace.net/2025/understanding-numpys-einsum/) derives carefully what the subscript string actually computes, and the [reference page](https://numpy.org/doc/stable/reference/generated/numpy.einsum.html) states the grammar precisely.

Side A closes with JAX, which completes the notation-to-computation arc: SymPy differentiates symbols, but `jax.grad` differentiates *programs*, and $\nabla f$ becomes a function you can call. The [automatic differentiation tutorial](https://docs.jax.dev/en/latest/automatic-differentiation.html) is the right entry, and [The Autodiff Cookbook](https://docs.jax.dev/en/latest/notebooks/autodiff_cookbook.html) goes as deep as you will ever need here — Jacobians forward and reverse, Hessians as forward-over-reverse — written so that the two modes of automatic differentiation stop being incantations and start being a choice you understand.

## Side B — computation becomes proof

The second side turns the direction around. Side A asked how to compute what the notation describes; Side B asks how to *check* what the notation claims, and it runs on one engine: the SMT solver.

Begin with the [Z3Py guide](https://ericpony.github.io/z3py-tutorial/guide-examples.htm), the standard hands-on walkthrough — build a solver, add constraints, call `check()`, read off a model — which carries you from arithmetic puzzles up to `ForAll` and `Exists`, the point where $\forall x\, \exists y$ stops being philosophy and becomes a query. Once the API is in your hands, the conceptual question arrives on schedule: what is the solver actually doing? The [Beginner's Tutorial on SMT](https://hanielbarbosa.com/papers/fm2024.pdf), written for FM 2024 by the people who build Z3 and cvc5, is the best single answer — the lazy architecture of a SAT solver consulting theory solvers, explained from the ground up. And [Programming Z3](https://theory.stanford.edu/~nikolaj/programmingz3.html), by Z3's own authors, sits between those two readings and beyond them; it is long, and it rewards being read slowly and in pieces, the way one reads a textbook rather than a blog post.

Why so much time on a solver? Because the verification triad that closes this mix — hypothesis, deal, and crosshair — is best understood as three interfaces to the same ambition, and the deepest of the three has Z3 underneath. The reading that ties them together comes first: Hillel Wayne's [Property Tests + Contracts = Integration Tests](https://www.hillelwayne.com/pbt-contracts/). His argument is the keystone of the repo's testing setup — once your functions carry contracts, "no input can violate any contract" becomes a single universal property, and your contracts quietly become test oracles. Read it before the tool docs and the three tools stop looking like separate choices.

Then take them in order of increasing ambition. Hypothesis tests properties by generating examples: the [tutorial introduction](https://hypothesis.readthedocs.io/en/latest/tutorial/introduction.html) explains `@given`, strategies, and shrinking, with the [quickstart](https://hypothesis.readthedocs.io/en/latest/quickstart.html) available when you would rather be running in five minutes; Wayne's [Property Testing with Complex Inputs](https://www.hillelwayne.com/post/property-testing-complex-inputs/) then takes on the two genuinely hard parts, finding good properties and generating structured inputs. Deal states the properties in the code itself: its [intro](https://deal.readthedocs.io/basic/intro.html) frames contracts as "typing on steroids" and walks the pre- and postcondition decorators. And CrossHair attempts what Hypothesis approximates — its [introduction](https://crosshair.readthedocs.io/en/latest/introduction.html) shows the workflow, and [How Does It Work?](https://crosshair.readthedocs.io/en/latest/how_does_it_work.html) reveals the trick that connects Side B back to its beginning: your function is called with symbolic values backed by Z3, so the solver you met three readings ago is now exploring execution paths no example generator would find. When you are ready to see how far that engine can be pushed, Philip Zucker's [Programming and Interactive Proving With Z3Py](https://www.philipzucker.com/programming-and-interactive-proving-with-z3py/) bends Z3 toward genuine theorem proving, and makes a fitting last track.

## Hidden track

NetworkX stands slightly apart from both sides — it neither executes notation nor checks claims, it just is the graph theory chapters' workbench. The [official tutorial](https://networkx.org/documentation/stable/tutorial.html) is short, and the reference [introduction](https://networkx.org/documentation/stable/reference/introduction.html) is the better conceptual read for one design decision that explains the whole API: a node can be any hashable Python object. Save both for the week a graphs chapter is open in the other tab — the same just-in-time principle the Foundations Mix is built on.
